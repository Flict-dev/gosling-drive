# Задача курсового проекта

Тема: серверная часть веб-приложения для хранения и обмена файлами.

Основной ориентир по масштабу и стилю отчета: файл `Текст.docx`.

Примерный масштаб, который нужно выдержать:

- полноценная серверная часть веб-приложения;
- REST API;
- база данных с несколькими связанными сущностями;
- аутентификация и авторизация;
- центральная бизнес-функциональность, подтвержденная тестами;
- базовый веб-интерфейс;
- Docker-сборка;
- отчет в стиле примера, оформленный по ГОСТ 7.32-2017.

## 1. Цель проекта

Разработать серверную часть веб-приложения для хранения, загрузки, скачивания и обмена файлами между пользователями.

Система должна позволять пользователям:

- регистрироваться и входить в приложение;
- загружать большие файлы через настоящий клиентский механизм загрузки по частям;
- хранить файлы в S3-совместимом объектном хранилище;
- просматривать свои файлы и папки;
- скачивать файлы;
- создавать публичные ссылки;
- выдавать доступ другим пользователям;
- работать с версиями файлов;
- видеть базовую статистику хранилища.

## 2. Технологический стек

Backend:

- Python;
- FastAPI;
- SQLAlchemy;
- Alembic;
- PostgreSQL;
- JWT-аутентификация;
- Pydantic;
- boto3 или совместимый S3-клиент.

Хранилище файлов:

- MinIO как локальный S3-совместимый storage;
- запуск MinIO в Docker;
- хранение файлов в bucket;
- хранение метаданных файлов в PostgreSQL.

Frontend / Web UI:

- серверные HTML-шаблоны или простой клиентский интерфейс;
- настоящий выбор файла через browser File API;
- разбиение больших файлов на части;
- отправка частей напрямую в MinIO по presigned URL;
- отображение прогресса загрузки;
- отмена или повтор загрузки неудачных частей.

Инфраструктура:

- Docker;
- Docker Compose;
- отдельные сервисы для backend, PostgreSQL и MinIO;
- переменные окружения через `.env`;
- миграции базы данных через Alembic.

Архитектурный подход:

- DDD;
- разделение на domain, application, infrastructure и presentation;
- бизнес-логика не должна быть размазана по API-роутерам.

## 3. Главная техническая особенность

Файлы не должны храниться в PostgreSQL.

PostgreSQL используется только для метаданных:

- владелец файла;
- имя файла;
- размер;
- MIME-тип;
- bucket;
- object key;
- статус загрузки;
- версии;
- права доступа;
- публичные ссылки;
- история операций.

Байты файлов должны храниться в MinIO.

Для больших файлов используется S3 Multipart Upload.

Общая схема загрузки:

1. Клиент отправляет backend запрос на начало загрузки.
2. Backend проверяет пользователя, права, размер и квоту.
3. Backend создает multipart upload в MinIO.
4. Backend возвращает клиенту `upload_id` и presigned URL для частей.
5. Клиент режет файл на части и отправляет их напрямую в MinIO.
6. Клиент передает backend список загруженных частей и их `ETag`.
7. Backend завершает multipart upload в MinIO.
8. Backend сохраняет финальные метаданные файла в PostgreSQL.

Важно: большие файлы не должны идти через backend целиком.

Правильный поток байтов:

```text
Browser -> MinIO
```

Неправильный поток байтов:

```text
Browser -> Backend -> MinIO
```

## 4. Основные сущности базы данных

Планируемые сущности:

- `User` - пользователь системы;
- `Folder` - папка пользователя;
- `File` - логическая запись файла;
- `FileVersion` - версия файла в объектном хранилище;
- `UploadSession` - сессия multipart-загрузки;
- `ShareLink` - публичная ссылка на файл;
- `AccessGrant` - выданный доступ другому пользователю;
- `AuditLog` - журнал действий.

Этого достаточно, чтобы проект был сопоставим по масштабу с примером из `Текст.docx`, где было несколько связанных моделей и центральная бизнес-функция.

## 5. Функциональные требования

### 5.1 Пользователи и безопасность

- регистрация пользователя;
- вход пользователя;
- получение JWT-токена;
- получение данных текущего пользователя;
- защита операций изменения данных;
- разграничение прав владельца, обычного пользователя и администратора.

### 5.2 Папки

- создание папки;
- просмотр списка папок;
- вложенные папки;
- переименование папки;
- удаление папки;
- просмотр содержимого папки.

### 5.3 Файлы

- создание записи о файле;
- загрузка файла в MinIO;
- просмотр списка файлов;
- просмотр информации о файле;
- скачивание файла;
- переименование файла;
- перемещение файла между папками;
- удаление файла.

### 5.4 Загрузка больших файлов

- начало multipart-загрузки;
- генерация presigned URL для частей;
- загрузка частей с клиента напрямую в MinIO;
- параллельная загрузка нескольких частей;
- отображение прогресса;
- завершение multipart-загрузки;
- отмена multipart-загрузки;
- сохранение статуса загрузки.

### 5.5 Версии файлов

- создание новой версии файла;
- просмотр списка версий;
- скачивание конкретной версии;
- хранение номера версии;
- отображение актуальной версии.

### 5.6 Обмен файлами

- создание публичной ссылки;
- отключение публичной ссылки;
- срок действия ссылки;
- ограничение количества скачиваний;
- скачивание файла по публичной ссылке.

### 5.7 Права доступа

- выдача доступа другому пользователю;
- права `read` и `write`;
- запрет доступа к чужим файлам;
- отзыв доступа.

### 5.8 Статистика и аудит

- общий объем файлов пользователя;
- количество файлов пользователя;
- количество скачиваний;
- журнал операций: upload, download, delete, share, access_grant;
- просмотр журнала администратором.

## 6. REST API

Планируемые группы endpoint-ов:

- `/api/auth/register` - регистрация;
- `/api/auth/login` - вход;
- `/api/auth/me` - текущий пользователь;
- `/api/folders/` - работа с папками;
- `/api/files/` - работа с файлами;
- `/api/files/{file_id}/download-url` - получение ссылки на скачивание;
- `/api/files/{file_id}/versions` - версии файла;
- `/api/uploads/initiate` - начало multipart-загрузки;
- `/api/uploads/{upload_id}/parts` - получение ссылок для загрузки частей;
- `/api/uploads/{upload_id}/complete` - завершение загрузки;
- `/api/uploads/{upload_id}/abort` - отмена загрузки;
- `/api/shares/` - публичные ссылки;
- `/api/public/{token}` - публичное скачивание;
- `/api/access/` - выдача и отзыв доступа;
- `/api/storage/stats` - статистика хранилища;
- `/api/audit/` - журнал действий.

## 8. Docker-сборка

Нужно подготовить `docker-compose.yml` со следующими сервисами:

- `backend` - FastAPI-приложение;
- `postgres` - PostgreSQL;
- `minio` - S3-совместимое хранилище;
- возможно, `minio-init` - создание bucket при первом запуске.

MinIO должен быть доступен:

- API: `http://localhost:9000`;
- Console: `http://localhost:9001`.

Backend должен быть доступен:

- API/Web: `http://localhost:8000`;
- Swagger/OpenAPI: `http://localhost:8000/docs`.

## 12. Критерий готовности

Проект можно считать готовым, если:

- приложение запускается через Docker Compose;
- backend доступен на `localhost:8000`;
- PostgreSQL работает в отдельном контейнере;
- MinIO работает в отдельном контейнере;
- пользователь может загрузить большой файл по частям;
- файл реально попадает в MinIO;
- метаданные файла появляются в PostgreSQL;
- файл можно скачать;
- публичная ссылка работает;
- чужие файлы защищены правами доступа;
- web-интерфейс позволяет выполнить основные действия;
- API документировано через Swagger;
- есть тестовые сценарии;
- подготовлен отчет в стиле `Текст.docx` по ГОСТ.
